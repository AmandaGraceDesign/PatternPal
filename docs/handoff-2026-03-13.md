### Known Bug: Ruler unit conversion is inaccurate

**Problem:** When toggling the ruler to `px` or `cm`, the displayed values don't correspond to the correct converted measurements. For example, if the ruler shows `24in` at a position, switching to `px` shows `400px` at that same position — but 24 inches at the file's DPI should be a different pixel count (e.g., 24 * 300 = 7200px at 300 DPI).

**Root cause analysis:**

The issue is in how `pixelsPerUnit` is calculated in `PatternPreviewCanvas.tsx` (lines 440-448) and how `formatLabel()` in `Ruler.tsx` (lines 156-166) uses it:

- `pixelsPerUnit` for px mode is calculated as `(96 / dpi) * viewZoom` — this represents how many screen pixels equal one pattern pixel
- But in `formatLabel()`, the px label is computed as `Math.round(value)` where `value` is the screen pixel position — it's not dividing by `pixelsPerUnit` to get the actual pattern pixel count
- For cm mode, `formatLabel()` does `value / pixelsPerUnit * 2.54`, but `pixelsPerUnit` for cm is `pixelsPerInch / 2.54`, so the 2.54 conversion is applied twice

**Where to fix:**
- `src/components/canvas/Ruler.tsx` — `formatLabel()` function (line 156)
- Possibly `src/components/canvas/PatternPreviewCanvas.tsx` — `getPixelsPerUnit()` (line 444)

**Key requirement:** Conversions must use the **file's DPI** (e.g., 300 DPI), not the screen DPI (96). If a position is at 24 inches, px mode should show `24 * 300 = 7200px` (not `24 * 96 = 2304px`), and cm mode should show `24 * 2.54 = 60.96cm`. Currently `PatternPreviewCanvas.tsx` calculates `pixelsPerPixel` using screen DPI (`96 / dpi * viewZoom`) which breaks the conversion.
